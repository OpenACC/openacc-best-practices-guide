OpenACCを使ったアプリケーションの高速化
----------------------------------------
このセクションでは、OpenACCを使用してアプリケーションを高速化するための段階的なアプローチについて詳しく説明します。このアプローチを取る際は、各ステップを複数回見直し、各ステップの結果が正しいかどうかを確認することが有益です。段階的に作業することで、各変更の範囲を限定し、生産性とデバッグを向上させることができます。

### OpenACCディレクティブ構文 ###
このガイドでは、移植プロセスで必要になるにつれて、OpenACCディレクティブを段階的に紹介していきます。すべてのOpenACCディレクティブには共通の構文があり、`acc`センチネルによってコンパイラに後続のテキストがOpenACCであることを示し、ディレクティブとそのディレクティブに対する句が続きます。これらの句の多くはオプションですが、コンパイラに追加情報を提供します。

CとC++では、これらのディレクティブはプラグマの形式を取ります。以下のコード例は、追加の句なしでOpenACCの`kernels`ディレクティブを示しています。

~~~~ {.c .numberLines}
    #pragma acc kernels
~~~~

Fortranでは、ディレクティブは特別なコメントの形式を取ります。以下に示すとおりです。

~~~~ {.fortran .numberLines}
    !$acc kernels
~~~~

一部のOpenACCディレクティブは構造化されたコードブロックに適用され、他のディレクティブは実行可能な文です。CとC++では、コードブロックは中括弧（`{`と`}`）で表現できます。Fortranでは、コードブロックはOpenACCディレクティブ（`!$acc kernels`）で始まり、対応する終了ディレクティブ（`!$acc end kernels`）で終わります。


### 移植サイクル ###
プログラマーは、正確性を確保するために、OpenACCを使用してアプリケーションを高速化する際に段階的なアプローチを取るべきです。このガイドでは、まずアプリケーションのパフォーマンスを評価し、次にOpenACCを使用してコード内の重要なループを並列化し、次にホストとアクセラレータ間の不要なデータ移行を削除するためにデータ局所性を最適化し、最後に特定のアーキテクチャでのパフォーマンスを最大化するためにコード内のループを最適化するというアプローチに従います。このアプローチは多くのアプリケーションで成功を収めています。これは、最大の成果をもたらす可能性の高い変更に優先順位を付けることで、プログラマーが迅速かつ生産的に高速化を実現できるためです。

各ステップを詳しく説明する前に、2つの重要な点に注意する必要があります。第一に、このプロセスの途中でアプリケーションのパフォーマンスが実際に低下する場合があります。初期の努力がパフォーマンスの低下をもたらしても、開発者は落胆すべきではありません。後で説明しますが、これは一般的にホストとアクセラレータ間の暗黙的なデータ移動の結果であり、移植サイクルの一部として最適化されます。第二に、各変更後にプログラムの結果の正確性を確認することが重要です。頻繁な正確性チェックにより、エラーが複合化する前に、すぐに発見して修正できるため、多くのデバッグ作業を節約できます。一部の開発者は、各成功した変更後にソースバージョン管理ツールを使用してコードのスナップショットを取ることが有益だと感じるかもしれません。これにより、破壊的な変更を迅速に破棄し、コードを既知の良好な状態に戻すことができます。

#### アプリケーションパフォーマンスの評価 ####
アプリケーションを高速化し始める前に、アプリケーションがどのルーチンやループで大半の時間を費やしているのか、そしてその理由を理解することが重要です。高速化の利益を最大化するためには、アプリケーションの最も時間のかかる部分を理解することが重要です。アムダールの法則は、並列アクセラレータでアプリケーションを実行することで達成可能な速度向上は、残りのシリアルコードによって制限されることを教えてくれます。言い換えれば、アプリケーションは、できるだけ多くのコードを高速化し、最も時間のかかる部分に優先順位を付けることで、最大の利益を得ることができます。コードの重要な部分を特定するために、簡単なアプリケーションタイマーを含む様々なツールを使用できます。

#### ループの並列化 ####
コードの重要な領域が特定されたら、OpenACCディレクティブを使用して、ターゲットデバイスでこれらの領域を高速化する必要があります。コード内の並列ループは、OpenACCコンパイラがターゲットアーキテクチャ用にコードを並列化するために必要な情報を提供するために、OpenACCディレクティブで装飾する必要があります。

#### データ局所性の最適化 ####
CPU + GPUアーキテクチャなどの多くの高速化アーキテクチャは、*ホスト*と*デバイス*に別個のメモリスペースを使用するため、コンパイラが両方のメモリでデータを管理し、正しい結果を保証するために2つのメモリ間でデータを移動する必要があります。コンパイラはアプリケーションの完全な知識を持っていることはめったにないため、正確性を確保するために慎重である必要があり、多くの場合、実際に必要以上にアクセラレータとの間でデータをコピーすることになります。プログラマーは、メモリができるだけ長くアクセラレータにローカルのままで、絶対に必要な場合にのみ2つのメモリ間で移動されるように、コンパイラにメモリの管理方法に関する追加情報を提供できます。プログラマーは、このステップでデータ移動を最適化した後に、最大のパフォーマンス向上を実感することがよくあります。

#### ループの最適化 ####
コンパイラは、内部ヒューリスティックとアプリケーションに関する限られた知識に基づいて、コード内の並列性をターゲットアクセラレータにマッピングする方法について決定を下します。コンパイラにより多くの情報を提供することで、並列性をアクセラレータにマッピングする方法についてより良い決定を下せるようにし、追加のパフォーマンスを得られる場合があります。従来のCPUアーキテクチャからGPUなどのより並列なアーキテクチャに移行する場合、アクセラレータの追加の並列性を公開するため、またはデータ移動の頻度を減らすために、ループを再構築する必要がある場合もあります。多くの場合、並列アクセラレータのパフォーマンス向上を動機として行われたコードリファクタリングは、従来のCPUにも有益です。

### ヘテロジニアスコンピューティングのベストプラクティス ###
多くのアプリケーションは、コード内に並列性がほとんど、あるいはまったく公開されていない状態で書かれています。並列性を公開しているアプリケーションは、少数のスレッドまたはプロセスが長時間実行され、それぞれが大量の作業を計算する、粗粒度の方法で行うことが頻繁にあります。しかし、最新のGPUや多コアプロセッサは、短命で最小限の作業を実行する細粒度のスレッドを実行するように設計されています。これらの並列アーキテクチャは、より多くの並列性を優先してシングルスレッドのパフォーマンスをトレードオフすることで、高いスループットを実現します。これは、ハードウェアの並列性の増加を考慮して設計されたOpenACCでアプリケーションを高速化する場合、大量のデータ再利用を伴う密にネストされたループを優先するようにコードをリファクタリングする必要がある場合があることを意味します。多くの場合、これらの同じコード変更は、キャッシュの使用とベクトル化を改善することで、より従来のCPUアーキテクチャにも利益をもたらします。

OpenACCは、個別のメモリを持つデバイスや、ホストと共有されるメモリスペースを持つデバイスでアプリケーションを高速化するために使用できます。共有メモリを利用するデバイスでも、アクセラレータ用の高速で近接したメモリと、ホストが使用するより大きく遅いメモリの階層が依然として存在することが頻繁にあります。このため、基盤となるアーキテクチャが個別のメモリを使用するか統合メモリを使用するかに関係なく、配列の再利用を最大化するようにアプリケーションコードを構造化することが重要です。OpenACCで使用するためにコードをリファクタリングする場合、開発しているデバイスが統合メモリを持っていても、個別のメモリを想定することが頻繁に有益です。これにより、リファクタリングにおいてデータ局所性が主要な考慮事項となり、結果として得られるコードが階層的メモリを利用し、幅広いデバイスに移植可能であることが保証されます。

### ケーススタディ - ヤコビ反復

このガイド全体を通じて、高速化プロセスの各ステップを実証するために簡単なアプリケーションを使用します。最初のそのようなアプリケーションは、反復ヤコビソルバーを使用して2Dラプラス方程式を解きます。反復法は、2Dラプラス方程式のような楕円型PDEの解を、許容できる許容誤差内で近似するための一般的な技術です。この例では、各点がその隣接点の値の平均として値を計算する簡単なステンシル計算を実行します。計算は、2つの反復間の値の最大変化が許容レベル以下になるか、最大反復回数に達するまで反復を続けます。ドキュメント全体で一貫した比較のために、例は常に1000回反復します。C/C++とFortranの両方のメイン反復ループを以下に示します。

~~~~ {.c .numberLines startFrom="52"}
    while ( error > tol && iter < iter_max )
    {
        error = 0.0;

        for( int j = 1; j < n-1; j++)
        {
            for( int i = 1; i < m-1; i++ )
            {
                Anew[j][i] = 0.25 * ( A[j][i+1] + A[j][i-1]
                                    + A[j-1][i] + A[j+1][i]);
                error = fmax( error, fabs(Anew[j][i] - A[j][i]));
            }
        }

        for( int j = 1; j < n-1; j++)
        {
            for( int i = 1; i < m-1; i++ )
            {
                A[j][i] = Anew[j][i];
            }
        }

        if(iter % 100 == 0) printf("%5d, %0.6f\n", iter, error);

        iter++;
    }
~~~~    

---

~~~~ {.fortran .numberLines startFrom="52"}
    do while ( error .gt. tol .and. iter .lt. iter_max )
      error=0.0_fp_kind
  
      do j=1,m-2
        do i=1,n-2
          Anew(i,j) = 0.25_fp_kind * ( A(i+1,j  ) + A(i-1,j  ) + &
                                       A(i  ,j-1) + A(i  ,j+1) )
          error = max( error, abs(Anew(i,j)-A(i,j)) )
        end do
      end do
  
      do j=1,m-2
        do i=1,n-2
          A(i,j) = Anew(i,j)
        end do
      end do
  
      if(mod(iter,100).eq.0 ) write(*,'(i5,f10.6)'), iter, error
      iter = iter + 1
  
    end do
~~~~

各例の最も外側のループは、*収束ループ*と呼ばれます。これは、最大誤差許容値または反復回数に達することで答えが収束するまでループするためです。ループの反復が発生するかどうかは、前回の反復の誤差値に依存することに注意してください。また、`A`の各要素の値は、データ依存性として知られる前回の反復の値に基づいて計算されます。これらの2つの事実は、このループを並列に実行できないことを意味します。

収束ループ内の最初のループネストは、隣接要素の現在の値に基づいて各要素の新しい値を計算します。この新しい値を別の配列に格納する必要があることに注意してください。各反復が新しい値を自分自身に格納すると、各要素が計算される順序が最終的な答えに影響を与えるため、データ要素間にデータ依存性が存在します。一時配列に格納することで、`A`が更新される前に、すべての値が`A`の現在の状態を使用して計算されることを保証します。その結果、各ループ反復は他の反復から完全に独立しています。これらのループ反復は、任意の順序で、または並列に安全に実行でき、最終結果は同じになります。このループは、最大誤差値も計算します。誤差値は、新しい値と古い値の差です。2つの反復間の変化の最大量が許容誤差内にある場合、問題は収束したと見なされ、外側のループは終了します。

2番目のループネストは、`Anew`に計算された値で`A`の値を更新するだけです。これが収束ループの最後の反復である場合、`A`は最終的な収束値になります。問題がまだ収束していない場合、`A`は次の反復の入力として機能します。上記のループネストと同様に、このループネストの各反復は互いに独立しており、並列化しても安全です。
